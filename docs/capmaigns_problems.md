# Проблемы кампаний

1. Первична именно цель кампании, а не CTA. Это видно по самому продукту: оператор задаёт отдельную цель, шаблоны описаны через outcome, а CTA сейчас просто используется как суррогат для трекинга. Значит, текущая связка “цель = deeplink” была удобным сокращением для первой версии, но архитектурно это ошибка.

2. “Отправлять в локальное время пользователя” должно значить буквально локальное wall-clock время получателя. Если оператор ставит 18:00, пользователь в Мадриде получает в 18:00 по Мадриду, а пользователь в Сан-Паулу в 18:00 по Сан-Паулу. DST должен сохранять локальный час, а не UTC-якорь. Всё остальное для CRM-оператора будет восприниматься как баг.

3. Точных продуктовых объёмов в коде не видно. Рабочая гипотеза: систему уже надо проектировать минимум под десятки тысяч получателей на одну кампанию, 1-3 шага в journey и регулярные правки активных кампаний. Для таких объёмов заранее материализовать весь journey уже опасно.

4. Overview должен быть надёжной операторской консолью. Это почти наверняка продуктовое ожидание, потому что там уже есть KPI, quick views, статусы, прогресс и lifecycle-действия. Если экран не является source of truth, его надо радикально упростить. В текущем виде он выглядит как операционная панель, значит должен вести себя как операционная панель.

5. Сегменты в ближайшей итерации не должны быть живым runtime-источником аудитории. Самый безопасный путь сейчас: кампания должна исполняться из собственной сохранённой audience definition, а происхождение “взято из сегмента/шаблона” надо оставить только как metadata для UI. Если позже понадобится настоящая связка с сегментами, её надо делать уже через versioned reference, а не через текущую полу-символическую ссылку.

# Самые важные шаги, которые я бы реализовывал в первую очередь:

1. Развязать цель и CTA. В кампании должна появиться явная tracked goal, а кнопка/действие внутри push должна стать отдельным полем. Для существующих кампаний сделать миграцию из текущего поведения.

2. Починить семантику recurring-расписания. Хранить и считать его как расписание в локальном времени пользователя, а не как UTC-якорь с поздней подгонкой окна. До этого я бы вообще не расширял recurring-фичи.

3. Свести UI и backend к одному операторскому контракту. Одни и те же правила readiness, одно и то же поведение schedule, настоящий серверный фильтр для проблемных кампаний, и либо полноценный pause/resume, либо полное удаление paused из интерфейса.

4. Упростить live-контракт кампании. Всё, что не влияет на исполнение прямо сейчас, нужно убрать из runtime-модели или перевести в display-only metadata. Особенно это касается сегментов, provenance и “будущих” полей, которые сейчас создают ложное ощущение поддержанного сценария.

5. Перестроить планирование journey. Вместо материализации всех шагов вперёд планировать только ближайший допустимый шаг, а следующий создавать после завершения предыдущего и проверки goal/suppression. Если это слишком большой рефакторинг для одного этапа, хотя бы ввести cleanup superseded и лишних downstream rows.

6. Сделать overview реально достоверным. KPI должны быть либо честно глобальными и так подписаны, либо подчиняться фильтрам. Quick views должны работать на уровне backend, а не на уровне уже загруженной страницы.
